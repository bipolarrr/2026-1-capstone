# Unity 프로젝트 공통 코딩 규칙

## 목적
이 규칙은 Unity 프로젝트 전반에서 일관된 초벌 코드 품질을 유지하기 위한 공통 기준이다.  
목표는 **읽기 쉽고**, **책임이 분리되어 있으며**, **기존 프로젝트와 충돌 없이 통합 가능한 코드**를 작성하는 것이다.

---

## 가장 중요한 원칙

- 기존 코드의 **나쁜 스타일**(축약어 남발, 헝가리안 표기법, 책임이 섞인 구조)은 답습하지 않는다.
- 그러나 프로젝트와의 호환성을 위해, 이미 존재하는 **클래스 / 메서드 / 필드 / 이벤트 연결점 / 씬 이름 / 컴포넌트 참조 구조**는 반드시 확인하고 필요 시 재사용한다.
- 이미 같은 역할의 코드가 존재하면 새로 복제하지 말고, **재사용 / 확장 / 부분 수정** 중 가장 작은 변경으로 해결한다.
- 목표는 “기존 스타일 모방”이 아니라, **기존 프로젝트에 무리 없이 통합되는 읽기 쉬운 코드**를 만드는 것이다.

---

## 1. 역할 분리 규칙

- 클래스는 책임 단위로 분리한다.
- 하나의 클래스가 여러 역할을 동시에 맡지 않게 한다.
- 예시:
  - `Controller`: 씬 흐름, 상태 전이, 연결점 관리
  - `ButtonHandler`: 버튼 클릭 이벤트 처리
  - `Popup` 또는 `Panel` 관련 클래스: 열기/닫기 및 UI 상태 관리
  - `ViewBinder` 또는 UI 갱신 전용 클래스: UI 참조 갱신
- 모든 씬이 반드시 동일한 클래스 구성을 가질 필요는 없다.
- 다만 **책임이 섞이지 않도록 분리 원칙은 항상 유지**한다.

---

## 2. 네이밍 규칙

- 클래스명, 메서드명, 프로퍼티명은 `PascalCase`
- private serialized field는 `camelCase`
- 헝가리안 표기법 금지
- 과도한 축약어 금지
- 이름만 보고 역할이 드러나야 한다

### 나쁜 예
- `ctrl`
- `btn`
- `mgr`
- `cg`
- `obj`
- `popupRef`

### 좋은 예
- `mainMenuController`
- `settingsPopup`
- `introDuration`
- `resultPanel`
- `playerStatusGroup`

---

## 3. 기존 코드와의 통합 규칙

- 기존 public API가 있으면 먼저 확인하고 재사용한다.
- 이미 존재하는 메서드와 동일한 역할을 하는 메서드를 새로 만들지 않는다.
- 기능이 겹치면 아래 우선순위를 따른다:
  1. 기존 메서드 호출
  2. 기존 클래스 확장
  3. 최소 범위 수정
  4. 정말 필요할 때만 새 구현 추가
- 기존 씬, 프리팹, 인스펙터 연결 구조와 충돌하지 않게 작성한다.
- 무조건 새 구조를 강요하지 말고, 현재 프로젝트에 자연스럽게 붙도록 작성한다.

---

## 4. 필드 선언 규칙

- Inspector 연결 대상은 기본적으로 `[SerializeField] private`를 사용한다.
- public field는 지양한다.
- 관련 필드는 `[Header("...")]`로 묶는다.
- Header 이름은 사람이 읽기 쉬운 자연어로 작성한다.
- 상수는 `private const` 또는 `private static readonly`를 상황에 맞게 사용한다.

---

## 5. 메서드 작성 규칙

- 메서드는 짧고 역할이 명확해야 한다.
- 하나의 메서드는 한 가지 일만 하게 한다.
- 길어지는 로직은 private helper method로 분리한다.
- Unity 생명주기 메서드는 필요한 경우에만 사용한다.
- `Awake`, `Start`, `OnEnable` 등에 모든 로직을 몰아넣지 않는다.
- 외부/UI 이벤트 진입용 public 메서드는 짧게 유지하고, 실제 동작은 내부 메서드로 위임한다.

---

## 6. null 처리 규칙

- Inspector 참조형은 항상 null 가능성을 고려한다.
- null 처리 방식은 의도가 드러나야 한다.
  - 동작 생략이 자연스러운 경우: `return`, `yield break`
  - 연결 누락을 알려야 하는 경우: `Debug.LogWarning(...)`
- null 참조를 무시한 채 위험하게 진행하지 않는다.

---

## 7. UI 코드 규칙

- UI 흐름은 읽는 사람이 순서를 바로 따라갈 수 있게 작성한다.
- 표시/비표시, 알파 조정, 상호작용 잠금, 팝업 열기/닫기, 탭 전환, 버튼 클릭 등은 메서드 이름만 보고 역할이 드러나야 한다.
- UI 참조는 의미 단위로 그룹화한다.
- 화면 진입 연출, 패널 전환, 버튼 동작은 난잡하게 섞지 않고 단계별로 나눈다.

---

## 8. 씬 흐름 규칙

- 씬별 흐름은 씬의 목적에 맞게 설계한다.
- 모든 씬이 같은 구조를 복붙할 필요는 없다.
- 다만 아래 원칙은 유지한다:
  - 진입 시 초기화
  - 사용자 입력 진입점 분리
  - 씬 전환 / 상태 전이 연결점 분리
  - 특정 UI 전용 로직은 해당 UI 담당 클래스에 배치

---

## 9. 로그 규칙

- 임시 연결점은 `Debug.Log(...)`를 사용할 수 있다.
- 누락 / 비정상 연결은 `Debug.LogWarning(...)`를 사용한다.
- 로그 메시지는 짧고 목적이 분명해야 한다.
- 에러가 아닌 상황에 `Debug.LogError(...)`를 남발하지 않는다.

---

## 10. 주석 및 문서화 규칙

- 모든 클래스 상단에는 XML summary 주석을 작성한다.
- 외부에서 호출되는 public 진입 메서드는 필요 시 summary를 붙인다.
- 주석은 “왜 필요한지” 또는 “무엇을 하는지”를 설명한다.
- 코드만 봐도 obvious한 내용을 장황하게 반복하지 않는다.

---

## 11. 초벌 작성 원칙

- 초벌 단계에서는 명확성과 유지보수성을 최우선으로 한다.
- 과도한 추상화, 불필요한 인터페이스, 성급한 일반화 금지
- 아직 정해지지 않은 미래 기능을 멋대로 확장 구현하지 않는다.
- 복잡한 시스템은 연결점만 명확히 남기고 `TODO`로 처리할 수 있다.
- 컴파일 가능성과 구조의 선명함을 우선한다.

---

## 12. 금지 사항

- 기존 코드의 나쁜 스타일을 그대로 답습하지 않는다.
- 의미 없는 축약형 변수명 사용 금지
- 하나의 클래스에 `Controller + Input + Animation + Popup + SceneLoading`을 모두 몰아넣지 않는다.
- Inspector 연결이 필요한 것을 public field로 대충 열어두지 않는다.
- 이미 있는 기능을 확인하지 않고 중복 구현하지 않는다.
- 미정인 기능을 마음대로 완성된 척 구현하지 않는다.

---

## 출력 방식

- 먼저 설계한 책임 분리를 짧게 설명한다.
- 그 다음 파일별로 나누어 코드를 출력한다.
- 각 파일은 바로 Unity 프로젝트에 넣을 수 있는 형태여야 한다.
- 컴파일 가능성을 우선한다.
- 아직 외부 시스템이 없는 부분은 `TODO` 또는 명확한 연결점 메서드로 남긴다.

---

## 최종 지시

- 기존 코드의 **나쁜 스타일은 따르지 말 것**
- 기존 프로젝트의 **심볼과 연결점은 존중하고 재사용할 것**
- 같은 기능이 이미 있으면 **복제하지 말고 최소 수정으로 통합할 것**
- 목표는 **새 코드 작성 자체**가 아니라, **좋은 구조로 안전하게 붙는 코드 작성**이다
