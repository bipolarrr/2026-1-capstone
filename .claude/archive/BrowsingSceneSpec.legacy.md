# BrowsingSceneSpec

이 문서는 Claude Code가 이번 작업에서 구현 범위와 금지선을 헷갈리지 않게 하기 위한 최소 스펙이다.  
세부 연출을 과하게 늘리지 말고, 아래 요구사항만 정확히 만족시키는 방향으로 구현한다.

## 1. 전제

- 반드시 [CLAUDE.md](C:\Users\song\desktop\Capstone\CLAUDE.md)의 규칙을 따른다.
- 씬은 `.unity` 파일을 수동 편집하지 말고 `Assets/Editor/`의 SceneBuilder로 생성한다.
- 기존 `DiceTest` 씬/빌더를 유지하는 방향으로 가지 말고, 필요한 런타임 로직과 씬 요소만 추려서 새 구조로 옮긴다.
- 새 씬은 아래 2개다.
  - `GameExploreScene`: 탐험 / 조우 / 아이템 선택
  - `GameBattleScene`: 일반몹 전투 / 보스 전투

## 2. 이번 작업의 핵심 흐름

- 메인 메뉴 → 캐릭터 선택 → `GameExploreScene`
- `GameExploreScene`에서 선형적으로 전진하다가 이벤트를 만난다.
- 이번 구현에서 이벤트는 아래 3종만 있으면 된다.
  - 일반 몹 전투
  - 아이템 상자
  - 보스 전투
- 기본 진행 순서는 고정으로 `일반 몹 전투 -> 아이템 상자 -> 보스 전투` 로 한다.
- 각 이벤트 해결 후 다시 탐험으로 복귀한다.
- 보스 전투 종료까지 한 사이클이 자연스럽게 이어지면 된다.

## 3. 데이터 / 매니저

- 지금부터는 `CharacterSelectionContext`만으로 부족하므로, 게임 전반에서 유지되는 정적 매니저를 추가한다.
- 이름은 꼭 `PlayerManager`일 필요는 없지만, 역할은 아래를 반드시 포함한다.
  - 현재 선택된 캐릭터 종류
  - 현재 플레이어 체력
  - 현재 보유 강화(파워업)
- 이번 실제 전투 구현은 `CharacterType.Dice`만 지원하면 된다.
- 단, 구조는 이후 `Mahjong`, `Holdem`도 같은 흐름에 붙일 수 있게 만든다.
- 전투 진입 시 어떤 적과 싸우는지, 보스전인지 여부, 선택 대상 정보도 전달 가능해야 한다.

## 4. 탐험 씬 요구사항

- 플레이어는 "앞으로 이동 중"처럼 보이는 간단한 연출이 있으면 된다.
  - 실제 복잡한 플랫포머 조작은 필요 없다.
  - 좌우/전진 느낌의 간단한 이동, 혹은 배경/캐릭터 이동 연출이면 충분하다.
- 이벤트를 만나면 이동이 멈추고 해당 UI를 띄운다.
- 현재 에셋이 없으므로 플레이어/적 비주얼은 아래 규칙으로 통일한다.
  - 파스텔톤 도형
  - 같은 계열의 더 진한 텍스트 색
  - 도형 위 또는 내부에 이름 표시
- 나중에 sprite/프리팹/이미지로 교체하기 쉽게, 표시용 컴포넌트와 실제 게임 데이터는 분리한다.

## 5. 일반 몹 전투 진입 규칙

- 일반 몹은 2~4마리 등장한다.
- 각 몹 체력은 5~20 범위에서 설정한다.
- 전투 시작 전, 플레이어가 우선 공격 대상을 지정해야 한다.
- 대상 지정 후 `공격`으로 `GameBattleScene`에 들어간다.
- 첫 롤 이전에는 `취소`로 탐험 씬의 대상 지정 상태로 돌아갈 수 있어야 한다.
- 전투 결과 적용 규칙:
  - 지정 대상: 최종 데미지 100%
  - 비지정 대상: 최종 데미지 30% 스플래시
    - 하드코딩 되어서는 안되며, 나중에 추가될 파워업에 의해서 수치 변경이 가능하게 구성되어야 한다.

## 6. 아이템 상자 규칙

- 상자를 열면 3개의 강화 선택지 중 1개를 고른다.
- 이번에는 전부 주사위 전투용 강화로 고정한다.
- 선택지는 아래 3개를 그대로 사용한다.
  - 홀수 눈만 또는 짝수 눈만으로 점수를 냈다면 데미지 2배
  - 모든 족보 데미지는 절반, 대신 족보가 아니면 데미지 2배
  - 일회성 부활 패시브: 다음 "사망에 이르는 데미지"를 1회 무효화하고 사라짐
- 효과는 실제 전투 계산에 반영되도록 설계한다.

## 7. 보스 규칙

- 현재 보스 조우 조건은 `일반 몹 전투 완료 후 아이템 상자 완료`로 고정한다.
- 보스 체력은 일단 `300`으로 한다.
- 수치는 하드코딩해도 되지만, 이후 스케일링 가능하게 한 곳에서 바꾸기 쉽게 둔다.
- 보스 전투도 `GameBattleScene`을 재사용한다.
- 보스의 "공격 전에 특정 주사위 패턴을 제시하고, 비슷하게 뽑을수록 피해 감소" 기믹은 이번에 완성하지 않아도 된다.
- 대신 이후 추가하기 쉽도록, 보스전 전용 UI/규칙을 붙일 수 있는 자리 정도만 남겨둔다.

## 8. 배틀 씬 요구사항

- `GameBattleScene`은 기존 `DiceTest`의 주사위 전투 코어를 가져와 재구성한다.
- 이번 구현에서 실제 전투 방식은 주사위 하나만 지원한다.
- 즉, `CharacterType.Dice`일 때만 정상 전투가 가능하면 된다.
- 최소 포함 요소:
  - 적 표시 영역
  - 대상 정보 또는 적 무리 정보
  - 주사위 5개
  - 최대 3회 굴리기
  - 홀드/언홀드
  - 예상 데미지 표시
  - 확정
  - 첫 롤 전 취소
- 기존 `YachtGameManager`, `YachtDie`, `DiceViewportInteraction`, `EnemyDisplay`는 재사용/분리/개명 가능하다.
- 다만 `DiceTest 전용 임시 구성`이 아니라 `GameBattleScene`의 런타임 구조로 정리한다.

## 9. 구현 범위 제한

- 이번에는 저장/로드, 인벤토리 화면, 복잡한 맵 분기, 실제 애니메이션 에셋 연동까지 가지 않는다.
- Mahjong/Holdem 전투 구현까지 하지 않는다.
- 보스 패턴 매칭 방어 기믹은 자리만 남기고 완성하지 않는다.
- ScriptableObject 중심의 과한 시스템화, DI, 과한 추상화는 하지 않는다.
- 현재 프로젝트 톤에 맞는 "직접적인 구조 + 필요한 정도의 확장성"만 확보한다.

## 10. Claude Code가 산출해야 하는 것

- 새 씬 2개용 SceneBuilder
- 탐험 씬 런타임 컨트롤러
- 배틀 씬 런타임 컨트롤러
- 전투 진입/복귀와 상태 전달에 필요한 최소 매니저/컨텍스트
- 기존 캐릭터 선택 이후 진입 씬 연결 수정
- `DiceTest` 의존 제거 또는 대체

## 11. 문서 해석 기준

- 애매하면 "이번 플레이 가능한 최소 수직 슬라이스"를 우선한다.
- 새 구조를 만들더라도 파일 수와 추상화는 최소한으로 유지한다.
- 겉보기에 그럴듯한 더미 구조보다, 실제로 탐험 → 전투 → 복귀 → 보스까지 이어지는 동작을 우선한다.
