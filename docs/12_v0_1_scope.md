# v0.1 Scope

Factual baseline:

- `docs/00_actual_project_audit.md`
- `docs/00_project_brief.md`
- `docs/01_architecture_map.md`
- `docs/02_unity_scene_and_object_construction.md`
- `docs/10_validation_triage.md`
- `docs/11_project_decisions.md`

This document defines v0.1 stabilization scope. It does not add new gameplay features.

Update note: the first-pass Hold'em implementation supersedes the earlier v0.1 assumption that Hold'em was placeholder-only. Hold'em is now source-level implemented, but still Unity-unvalidated until isolated Editor validation is completed.

## In Scope

- Dice battle playable loop.
- Mahjong battle playable loop.
- Hold'em first-pass playable loop, source-level only until Unity validation passes.
- Runtime flow:
  - `MainMenu`
  - `CharacterSelect`
  - `GameExploreScene`
  - `DiceBattleScene` / `MahjongBattleScene` / `HoldemBattleScene`
  - return to `GameExploreScene` or `MainMenu`
- Scene/build hygiene.
- Hybrid tracked/generated scene policy enforcement.
- Stale build settings cleanup for `SampleScene`, `YachtDice`, and `DiceTest`.
- EditMode validation baseline.
- Classification or correction of known failing EditMode tests.
- Asset pipeline policy documentation.
- Documentation cleanup needed to keep facts, decisions, and future ideas separate.

## Out Of Scope

- Slay the Spire-style node map.
- Shop system.
- Relic system.
- Joker system.
- Explicit Balatro-style shared chips/mult scoring model.
- Broad `SceneBuilderUtility` or SceneBuilder refactor.
- Broad controller rewrites.
- Asset deletion, asset moves, or `.meta` churn.
- Manual edits to generated `.unity` scene files.

## Acceptance Criteria

v0.1 is acceptable when all of the following are true:

1. A clean checkout has a documented path to open and validate required runtime scenes under the accepted Hybrid scene policy.
2. Stable runtime scenes are either tracked or have a documented regeneration procedure.
3. Experimental, obsolete, and temporary scenes are not required for runtime build/play validation.
4. `ProjectSettings/EditorBuildSettings.asset` no longer contains stale required entries for `SampleScene`, `YachtDice`, or `DiceTest`, or those entries are explicitly documented as intentional placeholders with owner and validation behavior.
5. Dice battle can be reached from the main flow, played through a battle, and return to explore or main menu as appropriate.
6. Mahjong battle can be reached from the main flow, played through a battle, and return to explore or main menu as appropriate.
7. Hold'em battle can be reached from the main flow, played through a battle, and return to explore or main menu as appropriate after scene generation.
8. Player defeat routes to main menu or the intended restart path without hidden scene dependencies.
9. Final stage victory routes to main menu or the intended victory flow without hidden scene dependencies.
10. EditMode tests are either passing or each remaining failure has a current documented classification, owner, and accepted stop condition.
11. Unity validation runs only in an isolated validation worktree and writes logs/results/build outputs outside the project root.
12. Asset source/intermediate/runtime categories are documented well enough that no asset cleanup task has to guess what is safe to delete.
13. README/project docs do not present out-of-scope systems as implemented v0.1 features.

## Manual QA Checklist

Run manual QA only after the scene/build hygiene task establishes the required runtime scene policy.

### Startup And Character Select

- Open the project in Unity `6000.3.11f1`.
- Confirm required stable scenes are present or regenerated by documented procedure.
- Start from `MainMenu`.
- Click play/start and reach `CharacterSelect`.
- Advance or skip cutscene slides.
- Select Dice and reach `GameExploreScene`.
- Return to main menu path still works where exposed.
- Select Mahjong and reach `GameExploreScene`.
- Select Hold'em and reach `GameExploreScene`.

### Explore Loop

- Confirm player HP and power-up UI initialize.
- Confirm normal combat encounter appears from the active stage.
- Confirm item-box encounter appears in the stage sequence.
- Confirm boss encounter appears at the expected stage point.
- Confirm `CurrentEventIndex` advances only through explore flow behavior.

### Dice Battle

- Enter `DiceBattleScene` from explore as Dice.
- Roll dice up to the current allowed roll count.
- Hold and unhold dice.
- Confirm score/attack flow damages enemies.
- Confirm enemy counterattack/defense phase completes.
- Confirm battle victory returns to `GameExploreScene`.
- Confirm cancel/defeat path returns to the intended destination.
- Confirm debug console commands used for QA do not corrupt session state.

### Mahjong Battle

- Enter `MahjongBattleScene` from explore as Mahjong.
- Draw and discard tiles.
- Confirm enemy wait/trigger behavior appears as currently designed.
- Confirm partial attack works.
- Confirm full win/yaku attack path works for at least one known hand if available.
- Confirm enemy attack/player damage can occur.
- Confirm battle victory returns to `GameExploreScene`.
- Confirm cancel/defeat path returns to the intended destination.

### Hold'em Battle

- Generate `HoldemBattleScene` from `Tools/Build HoldemBattle Scene` in the validation worktree unless it has been promoted as a tracked stable runtime scene.
- Enter `HoldemBattleScene` from explore as Hold'em.
- Confirm 1/3/5 community-card reveal stages.
- Redraw each hole card exactly twice.
- Use community redraw exactly once and confirm hole cards are preserved.
- Confirm High Card is single-target only.
- Confirm One Pair or better is AOE.
- Confirm enemy defense card minigame blocks equal/higher rank and fails lower rank.
- Confirm victory, cancel, defeat, and return paths.

### Victory, Defeat, Restart

- Complete a normal combat and return to explore.
- Complete a boss combat and observe stage/final victory behavior.
- Force or play into player defeat and confirm main menu/restart routing.
- Start a new game after defeat/victory and confirm session state resets.

### Build Hygiene

- Confirm build settings contain only intended runtime scene paths under the accepted policy.
- Confirm no missing required scene paths are silently present.
- Confirm build output is never written under `Assets/`.
- Confirm validation worktree status is clean or only contains explicitly allowed changes.

## Required EditMode Test Status

Before feature expansion:

- `BattleTests.EnemyAttackPositionResolverTests.Melee_StandsInFrontOfPlayer` must be fixed as test drift or reclassified with accepted rationale.
- `BattleTests.EnemyAttackPositionResolverTests.MidRange_StandsBetweenHomeAndMelee` must be fixed as test drift or reclassified with accepted rationale.
- `BattleTests.EnemyProjectileAttachmentFollowerTests.EnemyProjectileAttack_WithoutFollower_UsesShooterBodyFallbackStart` must be fixed as test drift or an explicit runtime first-frame behavior decision must be accepted.
- `MahjongTests.YakuEvaluatorTests.Toitoi_AllTriplets` must be resolved by deciding whether current closed-only all-triplet hands should assert `Suuankou`, support open/concealed metadata, or retain normal yaku alongside yakuman.

Required validation baseline:

- Focused failing tests from `docs/10_validation_triage.md` are corrected or intentionally waived.
- Full EditMode suite is run in a validation worktree.
- Any remaining failures are listed with exact test name, failure message, classification, owner, and stop condition.
- Validation command omits `-quit` for `-runTests` unless a newer verified command proves otherwise.
- Validation worktree status is checked after every run.

## Required Scene/Build Status

Required runtime scene set:

- `MainMenu`
- `CharacterSelect`
- `GameExploreScene`
- `DiceBattleScene`
- `MahjongBattleScene`
- `HoldemBattleScene`

Hybrid policy requirements:

- Each required runtime scene must be classified as tracked stable scene or generated scene with documented regeneration procedure.
- If a required scene is generated, the exact builder menu item or batch invocation procedure must be documented.
- Generated scenes must not be hand-edited.
- Stale build settings entries must be cleaned or explicitly marked as intentional non-runtime placeholders.
- Build settings must not require `SampleScene`, `YachtDice`, or `DiceTest` for v0.1 runtime validation unless a human owner restores them intentionally.

Required validation behavior:

- Unity tests/builds run in a separate worktree.
- Foreground checkout is never passed as `-projectPath` for automated validation.
- Logs, test results, and build outputs are outside the Unity project root.
- Build validation passes explicit `-buildTarget StandaloneWindows64`.
- Validation does not use `-ignorecompilererrors`.
- Validation does not use `-rebuildLibrary` on the foreground checkout.

## Explicit Non-Goals

- Do not add a node map.
- Do not add shop/relic/joker systems.
- Do not add a shared Balatro-style chips/mult scoring architecture.
- Do not broadly split scene builders or `SceneBuilderUtility`.
- Do not rename scenes, assets, serialized fields, public callback methods, or namespaces as part of v0.1 stabilization unless a specific task explicitly authorizes it.
- Do not delete MP4s, raw frames, rembg outputs, PNGs, `.meta` files, or other assets in baseline tasks.
- Do not modify generated scenes manually.
- Do not make Codex rely on untracked local-only scene files.
