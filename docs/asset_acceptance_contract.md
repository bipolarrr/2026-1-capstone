# Sprite Asset Acceptance Contract

This contract defines when generated sprite/video/frame assets are acceptable for this Unity project. It records the desired pipeline state only; it does not authorize asset moves, deletions, imports, `.meta` edits, scene changes, builder changes, or runtime changes.

## Source Of Truth Policy

For v0.1, do not delete any existing asset category. Treat these as source-of-truth candidates until the manifest is reviewed:

- Prompt/provenance documentation.
- Source/reference images used for Grok generation.
- Source/intermediate MP4s.
- Raw extracted frames.
- Transparent and transparent-clean outputs.
- Unity-ready PNGs currently loaded by builder or runtime paths.
- `.png.meta` sidecars generated by Unity.

Unity-ready PNGs are the immediate runtime dependency because builders and stage data load exact PNG paths and folders.

## Required Pipeline Stages

| Stage | Required state |
|---|---|
| Source image | Preserved when it is the prompt/reference input. Must be named so the character/action is clear. Unknown provenance is allowed but must be marked `unknown` in the manifest. |
| Source/intermediate MP4 | Preserved when it is the source for extracted animation frames. Record width, height, fps, frame count, action name, and whether it is still authoritative. |
| Raw extracted frames | Numbered `0.png` through `N-1.png`, no holes, dimensions consistent within the folder. Opaque RGB/RGBA is acceptable only at this stage. |
| Transparent output | Same numbering as raw frames, no holes, alpha channel present on every PNG, no fully empty-alpha frames. |
| Transparent-clean output | Same numbering as raw frames, no holes, alpha channel present, low-alpha background residue removed, consistent dimensions, no clipped subject unless intentionally full-frame. |
| Unity-ready PNG | The exact folder/file loaded by code. Must exist, have `.png.meta`, have stable numbering if animated, and satisfy the alpha/dimension/bounds checks below. |

## Frame Numbering

Animated folders must use direct numeric filenames:

- `0.png`
- `1.png`
- `2.png`
- ...
- `N-1.png`

Rules:

- No skipped frame numbers in the accepted runtime range.
- No duplicate semantic frames with nonnumeric names inside a Unity-ready runtime folder unless explicitly documented.
- If code calls `LoadNumberedPixelSprites(folder, "", "png", count)`, every frame from `0` through `count - 1` must exist.
- If code calls `LoadAvailablePixelSpriteFrames(folder)`, all direct `.png` files are loaded in trailing-number order; avoid unrelated PNGs in that folder.
- Intermediates may be nested, but the direct runtime folder must remain clean and predictable.

## Image Requirements

| Check | Acceptance rule |
|---|---|
| File existence | Every referenced PNG or referenced numbered frame must exist at the exact path used by code. |
| `.png.meta` sidecar | Every Unity-ready PNG must have a `.png.meta` sidecar. Do not edit it manually. |
| Dimensions | All frames in a single accepted animation folder must have identical width and height. |
| Alpha channel | Character, weapon, projectile, and tile sprites intended to overlay UI/backgrounds must have alpha. Opaque RGB is acceptable for full-screen backgrounds and raw extraction stages. |
| Non-transparent bbox | Every transparent frame must have a non-empty alpha bbox. Fully transparent frames are defective unless explicitly documented as intentional timing blanks. |
| Padding | The non-transparent bbox should not touch the image edge. Leave enough padding for motion and Unity tight-sprite import. Edge contact requires visual inspection and usually repadding. |
| Centering | Frame-to-frame subject center should not jump unless animation motion requires it. Large bbox center offsets require visual inspection. |
| Cropping | No body part, weapon, projectile, or effect should be cut off by the image boundary unless intentionally cropped for a full-frame background. |
| Resolution consistency | Do not mix dimensions inside one runtime animation folder. Cross-action dimensions may differ only when documented. |
| Background residue | Transparent-clean outputs should not retain low-alpha full-frame residue or matte pixels at image bounds. |

## MP4 Requirements

For every MP4 promoted as source/intermediate:

- Keep the MP4 next to the action it generated, or document the source location.
- Record width, height, fps, frame count, and intended output folder.
- Do not treat MP4 frame count alone as the runtime frame count; runtime count is determined by code and the direct Unity-ready PNG folder.
- If an MP4 has 145 frames but runtime intentionally uses fewer frames, document the trim policy.

## Folder Naming Contract

Observed current conventions:

- Runtime player folders: `Assets/Player/Sprites/<Action>/`
- Runtime enemy folders: `Assets/Mobs/Sprites/<Enemy>/<Action>/`
- Raw extracted frames: nested folder named after the MP4/action, for example `Player_Attack/`.
- Background removal output: `_transparent`.
- Cleaned background removal output: `_transparent_clean`.
- Size-normalized variants: suffixes such as `_764x640`, `_650x640`, or `_1000x1000`.

Acceptance:

- Do not normalize folder names in-place during audit/reporting tasks.
- New generated work should follow the observed names for the target action unless a human approves a new canonical convention.
- Do not place raw, transparent, and final runtime frames in the same direct folder unless that folder is explicitly the Unity-ready final.

## Runtime Reference Contract

Before marking an asset accepted:

1. Confirm whether it is directly loaded by code, stage data, builder code, or a ScriptableObject asset.
2. Confirm the exact path and case-sensitive filename.
3. Confirm folder loading mode:
   - exact file path;
   - numbered fixed count;
   - all available direct PNGs.
4. Confirm every referenced PNG has a sidecar `.png.meta`.
5. Confirm no path change would be required to use the asset.

## ScriptableObject And Tile Contract

`Assets/Mahjong/MahjongTileSprites.asset` currently references:

- `Assets/Mahjong/m_full.png`
- `Assets/Mahjong/p_full.png`
- `Assets/Mahjong/s_full.png`
- `Assets/Mahjong/w.png`

Acceptance rules:

- Do not replace or regenerate these files without preserving the intended sliced sprite references.
- Do not edit `MahjongTileSprites.asset` or `.meta` files during asset audit tasks.
- Any replacement tile sheet must be validated in Unity because sub-sprite file IDs matter.

## Defect Labels

Use these labels in future manifests and queues:

| Label | Meaning |
|---|---|
| `missing:file` | Exact referenced file does not exist. |
| `missing:folder-empty` | Referenced folder exists but contains no direct PNG frames. |
| `missing:frame-gap` | Numbered sequence has holes within the expected runtime range. |
| `defective:no-alpha` | Overlay sprite lacks alpha in a Unity-ready output. |
| `defective:empty-alpha` | Transparent frame has no visible pixels. |
| `defective:edge-contact` | Alpha bbox touches an image edge and may be cropped or contaminated. |
| `defective:residue` | Low-alpha or matte pixels make the bbox cover more area than the visible subject. |
| `defective:dimension-mismatch` | Frames in one accepted runtime folder have inconsistent dimensions. |
| `defective:count-mismatch` | Direct runtime folder count does not match a code-required count. |
| `unknown:unreferenced` | Asset is present but no scanned code or ScriptableObject reference was found. |

## Acceptance Checklist

For a generated animation set:

- [ ] Source image or prompt provenance is recorded.
- [ ] MP4 source is preserved or explicitly not applicable.
- [ ] Raw frames are present with contiguous numbering.
- [ ] Transparent-clean output is present with contiguous numbering.
- [ ] Unity-ready direct folder is identified.
- [ ] Runtime frame count matches the code expectation or is documented as available-frame loading.
- [ ] All direct runtime frames share dimensions.
- [ ] All direct runtime overlay frames have alpha.
- [ ] No fully transparent frames exist.
- [ ] Alpha bbox does not touch the edge, or a human visual decision accepts it.
- [ ] `.png.meta` exists for every Unity-ready PNG.
- [ ] No code or builder path change is needed to load it.

For a static sprite:

- [ ] Exact referenced path exists.
- [ ] `.png.meta` exists.
- [ ] Dimensions are suitable for current builder layout.
- [ ] Alpha is present for overlay sprites; opaque is acceptable for backgrounds.
- [ ] Non-transparent bbox is not accidentally full-frame because of residue.
- [ ] Any source/intermediate variant is classified in the manifest.

## Non-Goals

- No asset deletion.
- No asset moves or renames.
- No `.meta` edits.
- No Unity reimport.
- No scene or prefab edits.
- No ProjectSettings or Packages edits.
- No builder/runtime code changes.
