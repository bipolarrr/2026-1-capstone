# Project Brief

Factual baseline: `docs/00_actual_project_audit.md`. This brief describes only systems directly supported by the repository and current project docs.

## Current Proven Project Identity

This is a Unity 6 turn-based battle prototype where the player chooses a weapon style, enters a linear explore flow, and resolves encounters through implemented dice, mahjong, or first-pass Hold'em battle scenes.

Proven gameplay flow:

```text
MainMenu -> CharacterSelect -> GameExploreScene -> DiceBattleScene/MahjongBattleScene/HoldemBattleScene -> GameExploreScene/MainMenu
```

Stage progression is currently a linear stage-round structure using `StageData.rounds`, not a proven node-map system.

## Technology Stack

- Unity version: `6000.3.11f1` (`ProjectSettings/ProjectVersion.txt`)
- Runtime: C# / .NET Standard 2.1 per `AGENTS.md`
- Render pipeline: Universal Render Pipeline (URP)
- UI: UGUI, TextMesh Pro, Unity Input System UI module
- Testing: Unity EditMode tests under `Assets/Editor/Tests/`
- Target noted by project docs: Windows Standalone x64
- No CI/CD or linting is currently documented as active.

## Implemented Gameplay Loop

- `MainMenu` starts the flow.
- `CharacterSelect` shows cutscene slides and weapon choices.
- `GameExploreScene` advances through stage rounds.
- Combat rounds route to dice, mahjong, or Hold'em battle based on `GameSessionManager.SelectedCharacter`.
- Battle victory returns to explore; defeat or final victory routes to main menu.
- Item-box rounds grant one of the currently implemented `PowerUpType` values.

## Implemented Combat Modes

- Dice combat is implemented by `BattleSceneController`, `DiceRollDirector`, `DamageCalculator`, `DefenseCalculator`, `EnemyDiceRoller`, and related battle UI/animation classes.
- Mahjong combat is implemented by `MahjongBattleController` plus rule/model classes such as `TileWall`, `Hand`, `HandDecomposer`, `YakuEvaluator`, `BestHandPicker`, `MahjongDamageTable`, and `EnemyMahjongState`.
- Hold'em combat is source-level implemented by `HoldemBattleController` plus rule/model classes under `Assets/Scripts/Holdem/`. It routes to `HoldemBattleScene`, generated by `Tools/Build HoldemBattle Scene`. Unity validation is still required.

## Placeholder Or Unimplemented Systems

- A Slay-the-Spire-style node map is not proven. Current progression is a linear `StageRoundType` list.
- Shops, relics, jokers, deck-building systems, and card-modifier systems were not found as implemented systems.
- Balatro-like chips/mult scoring is not implemented as a shared scoring model. Dice and mahjong have separate scoring/damage systems.

## Scene Generation Model

The repository does not contain one class named `SceneBuilder`. Scene construction is split across `SceneBuilderUtility` and scene-specific editor builders:

- `Assets/Editor/MainMenuSceneBuilder.cs`
- `Assets/Editor/CharacterSelectSceneBuilder.cs`
- `Assets/Editor/GameExploreSceneBuilder.cs`
- `Assets/Editor/DiceBattleSceneBuilder.cs`
- `Assets/Editor/MahjongBattleSceneBuilder.cs`
- `Assets/Editor/HoldemBattleSceneBuilder.cs`

Generated scenes are not the source of truth. Builder scripts create objects, wire serialized fields through `SceneBuilderUtility.SetField()`, and register persistent button callbacks with `UnityEventTools.AddPersistentListener()`.

## Asset Pipeline Summary

Generated asset pipeline evidence exists:

- PNG frame folders under `Assets/Player/Sprites/` and `Assets/Mobs/Sprites/`
- MP4 source/intermediate files under player and mob sprite folders
- `_transparent`, `_transparent_clean`, and `_764x640` output folders
- Prompt/process notes in `docs/grok-imagine-sprite-prompts.md`
- Asset dependency notes in `docs/assets.md`

Extraction/background-removal scripts such as `extract_frames` or `rembg` were not found as tracked files in the inspected repo paths.

## Current Major Risks

- `Assets/Scenes/` contains generated scene artifacts whose tracking policy must stay aligned with build settings.
- Current source routing requires `HoldemBattleScene`, but the foreground `ProjectSettings/EditorBuildSettings.asset` inspected for this pass does not yet list it.
- Scene builders use string-based serialized-field wiring; field renames can silently break generated scenes unless builders and validation are updated.
- `HoldemBattleScene` must be generated or promoted with its `.meta` before play/build validation because Hold'em routing now targets it.
- Public callback methods wired through persistent listeners are scene contracts.
- Hard-coded asset paths in builders and stage data can break visual/audio wiring if assets move.
- Unity validation must be run only in an isolated validation worktree, not against the foreground checkout.

## What Not To Assume

- Do not assume Hold'em is Unity-validated until the scene is generated and isolated tests/build/play checks pass.
- Do not assume a node-map, shop, relic, joker, deck-building, or Balatro chips/mult system is implemented.
- Do not assume generated scenes are tracked source of truth.
- Do not assume extracted sprite assets can be reproduced from tracked scripts.
- Do not assume all EditMode tests currently pass until Unity validation is run in the required isolated worktree.
